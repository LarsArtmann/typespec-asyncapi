## g) TOP 2 QUESTIONS

### 1. Should `SecurityScheme.type` be strictly `SecuritySchemeType` (no `string` escape hatch)?

You said: "maybe we should remove the 'string' part instead." I agree — but this means the `$security` decorator must validate the `type` field at compile time and emit a diagnostic like `"Unsupported security scheme type 'xyz'. Supported: http, apiKey, oauth2, openIdConnect, mutualTLS, plain, scramSha256, scramSha512, gssapi, external, oauthBearer, X509, sasl"`. This is the right call for type safety, but it means users who pass custom/unknown security types will get compile errors. **Is that the behavior you want?**

### 2. Should I fix the channel-address-slash `$ref` bug now?

The new `$ref` chain test discovered that channel addresses containing `/` (e.g., `orders/events`) produce `$ref` values that standard JSON pointer resolvers cannot follow. This affects any consumer of the generated AsyncAPI spec that uses a standard JSON pointer resolver (like AsyncAPI parsers). Fixing it requires either URL-encoding `/` as `~1` in channel keys, or disallowing `/` in channel addresses entirely. **Which approach do you prefer?**
