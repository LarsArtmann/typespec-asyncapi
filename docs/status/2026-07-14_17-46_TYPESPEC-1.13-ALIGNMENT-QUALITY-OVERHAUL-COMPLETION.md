## G) TOP 2 QUESTIONS

### 1. How should the `compileAsyncAPI` vs `compileRaw` diagnostic split brain be fixed?

**Context:** `compileAsyncAPI` uses `tester.compile()` which catches exceptions but doesn't surface all decorator diagnostics. `compileRaw` uses `instance.compileAndDiagnose()` which surfaces all diagnostics but has a two-pass architecture (pass 1 for diagnostics, pass 2 for output) that can produce different results.

**Why I can't figure this out:** The TypeSpec `createTester` API is underdocumented. I don't know if `tester.compile()` is *supposed* to surface decorator diagnostics or if it's a known limitation. The two-pass approach in `compileRaw` may be a workaround for a TypeSpec bug, or it may be the correct usage pattern. I'd need to read the `@typespec/compiler` source or ask the TypeSpec team.

**Impact:** Until this is resolved, test helpers will remain duplicated and some tests will be fragile (depending on which API they use).

### 2. Should the channel-slash `$ref` bug be fixed by URL-encoding or by validation?

**Context:** Channel addresses like `orders/events` produce `$ref` values like `#/channels/orders/events` which are ambiguous — JSON pointer resolvers interpret this as `channels.orders.events` (two levels) instead of `channels["orders/events"]` (one key).

**Option A:** URL-encode `/` as `~1` in channel keys per RFC 6901. This preserves user intent (`orders/events` stays as one channel) but produces ugly `$ref` values like `#/channels/orders~1events`.

**Option B:** Validate and reject `/` in channel addresses with a diagnostic. This is simpler but forces users to use `orders.events` or `orders_events` instead.

**Why I can't figure this out:** This is a design decision. AsyncAPI 3.0 spec doesn't explicitly disallow `/` in channel addresses, but the JSON Pointer spec (RFC 6901) defines `/` as the path separator. Both options have tradeoffs that depend on how users actually use channel addresses in practice. I don't know if the AsyncAPI ecosystem (parsers, tools, code generators) handles `~1` in `$ref` keys correctly.
