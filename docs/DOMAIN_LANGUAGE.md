# Domain Language

**TypeSpec AsyncAPI Emitter** — A TypeSpec library that transforms TypeSpec definitions into AsyncAPI 3.1 specification documents (YAML or JSON).

## Core Concepts

### Channel

An addressable communication path (e.g., `user.events`, `orders/{orderId}`). Defined with `@channel("address")` on an operation. Maps to an AsyncAPI [Channel Object](https://www.asyncapi.com/docs/reference/specification/v3.1.0#channelObject).

### Operation

A publish or subscribe action on a channel. Defined with `@publish` or `@subscribe` decorators on TypeSpec operations. Maps to an AsyncAPI [Operation Object](https://www.asyncapi.com/docs/reference/specification/v3.1.0#operationObject).

### Message

A unit of data exchanged over a channel. Derived from the return type of an operation. Maps to an AsyncAPI [Message Object](https://www.asyncapi.com/docs/reference/specification/v3.1.0#messageObject).

### Server

A connection endpoint with a URL and protocol. Defined with `@server("name", #{ url: "...", protocol: "..." })`. Maps to an AsyncAPI [Server Object](https://www.asyncapi.com/docs/reference/specification/v3.1.0#serverObject).

### Schema

A JSON Schema representation of a TypeSpec model. Generated by the AssetEmitter and stored in `components.schemas`. Maps to an AsyncAPI [Schema Object](https://www.asyncapi.com/docs/reference/specification/v3.1.0#schemaObject).

## Decorators

| Decorator                             | Target                      | Purpose                              |
| ------------------------------------- | --------------------------- | ------------------------------------ |
| `@channel(address, description?)`     | Operation                   | Define channel address               |
| `@publish`                            | Operation                   | Mark as send operation               |
| `@subscribe`                          | Operation                   | Mark as receive operation            |
| `@server(name, config)`               | Namespace                   | Define server endpoint               |
| `@message(config)`                    | Model                       | Message metadata (contentType, etc.) |
| `@protocol(config)`                   | Operation, Model            | Protocol-specific binding            |
| `@security(config)`                   | Operation, Namespace        | Security scheme configuration        |
| `@tags([...])`                        | Model, Operation, Namespace | Categorization tags                  |
| `@correlationId(location, property?)` | Model                       | Correlation ID location              |
| `@bindings(config)`                   | Operation, Model            | Protocol-specific bindings           |
| `@header(name, value)`                | Model, ModelProperty        | Message header definition            |

## `$ref` Chain

The AsyncAPI 3.1 specification requires a specific reference chain:

```
operations → #/channels/{channelId}/messages/{messageId}
channels → #/components/messages/{messageId}
components.messages → #/components/schemas/{schemaName}
```

Operations MUST NOT reference `#/components/messages/` directly. This is a spec requirement, not a design choice.

## Emitter Flow

```
$onEmit(context)
  → consolidateAsyncAPIState(program)    // Read decorator state
  → generateSchemas(context)             // AssetEmitter → JSON schemas
  → buildAsyncAPIDocument(state, schemas, options)  // Assemble document
  → emitFile(program, content)           // Write YAML/JSON
```

## Protocols

Supported protocols (from `src/constants/protocols.ts`): `http`, `https`, `ws`, `wss`, `mqtt`, `mqtt5`, `kafka`, `amqp`, `amqp1`, `nats`, `jms`, `sns`, `sqs`, `stomp`, `redis`, `mercure`, `ibmmq`, `pulsar` (18 canonical names). Friendly aliases like `websocket` are accepted as input and normalized to `ws` via `normalizeProtocol()`.
