03Tier 1 — Foundation (1% → 51%)

Critical Path

Three tasks that deliver disproportionate value. The refactor unblocks ALL future document-builder work. The other two are trivial gaps that users notice immediately.

Task 1: Refactor buildAsyncAPIDocument()

Current state: 371 lines, 161 statements, complexity 84. ESLint violations: max-lines-per-function (limit 350), max-statements (limit 150), max-lines (limit 500). The function handles channel creation, operation discovery, message registration, server building, security schemes, schema ref wiring, and $ref chain construction — all in one function.

Target state: Extract into src/builders/ modules — channel-builder.ts, operation-builder.ts, message-builder.ts, server-builder.ts, security-builder.ts. The main function becomes a thin orchestrator. Each builder receives a shared BuilderContext.

Risk This is the most invasive refactor. Must be done BEFORE any Tier 2 document-builder work. Full test suite must pass with zero regressions.

Task 2: defaultContentType on document root

The field defaultContentType?: string already exists in the document model (asyncapi-document.ts:238) but is never populated. Add a @defaultContentType decorator, wire it into state, and emit it on the root document. Trivial — 30 minutes.

Task 3: @doc propagation to channels

@doc already works for models, properties, operations, and messages. Channels silently drop it. Fix: add getDoc() call when building ChannelObject in document-builder. 30 minutes including test.

04Tier 2 — Core Features (4% → 64%)

High Value

High-impact features that unblock real-world use cases, plus type safety cleanup that improves the entire test suite.

Task 4: Operation reply support

The OperationReply interface exists (asyncapi-document.ts:124) and the reply? field is on OperationObject — but it's never populated. Request-reply is a fundamental messaging pattern. Add a @reply decorator that specifies the reply message model. Wire into the operation builder. Must validate against the AsyncAPI 3.1 JSON Schema.

Task 5: Multi-message operations

Currently each operation maps 1:1 to one message. AsyncAPI supports operations referencing multiple message types (e.g., an event stream with different event schemas). Design: accept a union of Models on the operation, or a @messages decorator accepting an array. Update DiscoveredOp and the operation builder to emit a messages array instead of a single message reference.

Tasks 6–8: Type Safety Cleanup

  • Task 6: Remove 22 as AsyncAPIObject casts across 5 test files (test-helpers.ts, subscribe.test.ts, cli-test-helpers.ts, asyncapi-generation.test.ts, decorator-functionality.test.ts)
  • Task 7: Clean 111+ Record<string, unknown> casts in 6 compliance test files (protocol-bindings: 35, edge-cases: 21, schema-types: 19, servers-security: 13, ref-chain: 13, document-structure: 10)
  • Task 8: Fix schema-validator.ts and type-guards.ts to return/assert ParsedAsyncAPIDocument instead of Record<string, unknown>
Why now These are pure type-safety improvements. They make the test suite self-documenting and catch type mismatches at compile time instead of runtime.

Task 9: AsyncAPI Studio round-trip verification

If the emitter's output doesn't work in AsyncAPI Studio (the primary consumer tool), users lose trust immediately. Write a round-trip test: emit a representative spec with all features, parse it back with @asyncapi/parser, and verify structural integrity. Note: @asyncapi/parser v3.6.0 has Bun incompatibility (AJV new Function() codegen) — use Node.js for this test, or manual $ref resolution tests.

05Tier 3 — Completeness (20% → 80%)

Important

Protocol coverage, validation depth, and DX polish. These make the emitter "complete" for the 80% use case.

Tasks 10–11: Server Bindings & Explicit IDs

Task 10: @server currently emits host/protocol/description but no bindings. The binding framework already exists — extend it to server targets. Depends on Task 16 (@bindings for Namespace).

Task 11: @operationId and @messageId decorators for explicit naming control. Currently IDs are auto-generated from TypeSpec type names. Users need deterministic names for cross-file references and documentation.

Task 12: Binding field-level validation

Currently processBindings() validates keys and versions structurally. It does NOT validate individual field values against the AsyncAPI spec. Load @asyncapi/specs/bindings/{protocol}/ JSON Schemas and validate field values at decorator time. This catches errors like qos: 99 (MQTT allows 0–2) before the user sees a cryptic JSON Schema validation failure.

Foundation for Tasks 13–15 (new protocols). Do this before adding Redis, GCP Pub/Sub, and SNS so the validation framework is in place.

Tasks 13–15: New Protocol Bindings

Redis, Google Cloud Pub/Sub, and AWS SNS. Each follows the established pattern: add protocol constant, binding version, placement matrix entries, compliance tests. Specs available in node_modules/@asyncapi/specs/bindings/.

Tasks 16–18: Architecture & DX

  • Task 16: @bindings for Namespace target — enables server binding placement validation
  • Task 17: Consolidate ESLint and oxlint — resolve contradictory rules, document strategy
  • Task 18: Coverage tooling — vitest can't instrument dist/*.js; research and fix

09Risks and Mitigations

Risk: Refactor introduces regressions

The buildAsyncAPIDocument() refactor (Task 1) touches the core path. 555 tests must all pass. The $ref chain construction is subtle.

Mitigation Do the refactor as pure extraction — move code, don't change logic. Run the full compliance suite (78 tests against official JSON Schema) after each subtask. Golden file tests lock verified-correct output.

Risk: @asyncapi/parser Bun incompatibility

The AsyncAPI Studio round-trip test (Task 9) needs @asyncapi/parser, which fails under Bun due to AJV new Function() codegen.

Mitigation Run the round-trip test under Node.js, or use manual $ref resolution tests (as already done in semantic-ref-resolution.test.ts).

Risk: Multi-message design ambiguity

How should multiple messages map from TypeSpec? Union of models? Array decorator? The wrong choice creates a maintenance burden.

Mitigation Prefer TypeSpec-native patterns: a union of message Models is the idiomatic way. Only add a @messages decorator if the union approach can't express the needed semantics.

Risk: Coverage gate reliability

If vitest can't instrument dist/*.js, the 75% coverage gate may be unreliable, giving false confidence.

Mitigation Task 18 explicitly addresses this. Until fixed, treat coverage numbers as directional, not authoritative.
